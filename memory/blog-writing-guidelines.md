# Guidelines Rédaction Articles Blog Tech

## Règles pour tous les articles

### 1. Style d'écriture
- **Langage accessible** : Pas trop soutenu, éviter les tournures trop littéraires
- **Expliquer les termes techniques** : Quand on utilise un jargon (ex: "agentic coding"), expliquer brièvement ce que ça veut dire
- **Ton** : Professionnel mais conversationnel, comme une discussion entre experts qui expliquent à un curieux
- **Éviter** : Les phrases trop longues, les subordonnées complexes à rallonge

### 2. Structure
- **Introduction narrative** : Accrocher avec une histoire ou un contexte
- **Sections avec H2** : Titres clairs, pas trop de sections (5-7 max)
- **Pas de listes à puces interminables** : Préférer les paragraphes, utiliser les puces avec parcimonie
- **Conclusion avec une idée forte** : Pas un résumé en points, une vraie réflexion de fermeture

### 3. Mise en forme MDX
- Utiliser `##` pour les sections principales
- Utiliser `###` pour les sous-sections si nécessaire
- **Tableaux uniquement pour les comparatifs concrets** (prix, features, etc.)
- Code blocks avec langage précis (typescript, python, bash...)

### 4. Sources et actualité
- **TOUJOURS vérifier les dates** : Pas de "dernières nouveautés" de 2023 en 2026
- **Citer les sources** : Hugging Face, blogs officiels, etc.
- Faire des recherches web avant d'écrire sur un sujet tech

### 5. SEO
- Titre accrocheur mais descriptif
- Excerpt de 1-2 phrases qui résument le contenu
- Tags pertinents (3-5 max)

### 6. Public cible
Développeurs et curieux tech, mais pas que des experts. Il faut qu'un CTO puisse partager l'article à son équipe commerciale sans qu'elle soit perdue.

### Checklist avant publication
- [ ] Langage pas trop soutenu ?
- [ ] Termes techniques expliqués ?
- [ ] Phrases courtes et claires ?
- [ ] Sources vérifiables et récentes ?
- [ ] Pas trop de puces/listes ?
- [ ] Conclusion littéraire (pas un résumé en points) ?
